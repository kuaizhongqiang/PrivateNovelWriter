# Beta 待办清单

> 尚未讨论的议题，逐项处理、销项关闭。

---

## 1. 桌面端定位 — ✅ 已定

Desktop 有两个角色：

- **独立运行** — 当前 alpha 模式，自带 Agent B + 直连 DB，不依赖 Server
- **Server 客户端** — 连接远程 Server，作为桌面版的轻量前端

两者可共存，启动时选择。Desktop 不废弃，也不冻结。

---

## 2. 导出能力 — ✅ 已定

分两部分：

**阅读界面：**

- Gateway UI 中的一个阅读页面，类似小说网站风格
- 章节列表 → 点击跳转 → 阅读正文

**TXT 导出：**

- 分章节导出到文件夹（已有 `text/{phase_name}/ch-xxx.txt` 结构）
- 合并导出为一个 .txt 文件

EPUB/PDF 等暂不支持，beta 不做。

---

## 3. 安装与分发 — ✅ 已定

发布渠道：

- **GitHub Releases** — Rust 二进制发布，CI 完成编译和上传
- **npm** — 可选 wrapper 包，给非 Rust 用户（如 JS 插件生态）

Docker 待容器化方向正式启动时再做（P1），发布到 Docker Hub 或 ghcr.io。

---

## 4. 稳定性与错误处理 — ✅ 已定

**请求超时处理：**

- Agent A（外部工具调用）— 返回 task_id / status，Agent A 轮询结果
- Agent B（Server 内部 LLM 调用）— 阻塞等待，但 Agent A 可发取消指令关掉它

**LLM 调用失败重试：**

- 不自动重试。失败后告诉 Agent A，由 Agent A 决定重发或删除

**数据一致性保证：**

设计原则：**以 DB 为准**。脏文件允许存在，但需要有清理机制。

规划方向（后续细化）：

- 写入改为 temp → rename 原子模式（同文件系统 rename 是原子的）
- DB 先写入记录标记状态，再写文件，最后更新状态
- Server 启动时做一次孤儿文件/记录清理

**日志与可观测性：**

- ✅ Token 消耗和 cache 命中率必须记录
- ✅ 其他日志先做简单处理（stdout，分级输出）

---

## 5. Schema 升级兼容 — ✅ 已定

暂无外部用户，beta 发布时如有 schema 变更：

- 加 `schema_version` 表记录版本号
- Server 启动时检查，不一致则自动 migration 或提示
- `.txt` 正文文件不受影响

随 Server 开发一并处理。

---

## 6. 端到端用户文档 — ✅ 已定

目标读者是 Agent A，不是人。

- Skill 文档作为包内静态文件（`skill.md`），随代码一起发布
- Plugin 工具 description 中告知 Agent A：需要时到指定路径读取
- Server 端点不是必需的

不用单独写"用户手册"——Skill 文档本身就是 Agent A 的说明书。
